## Phase 1: Core Utilities & API Bugs

### Bug 1.1: Token Tracker Crash on Null Inputs (Critical)
*   **Severity:** Critical
*   **Component:** Token Tracking & PM Governance
*   **Affected File:** [governance.py](file:///e:/Mult_agent/agency/governance.py#L79-L88) (`TokenTracker.estimate_tokens`)
*   **Description:** The helper function `estimate_tokens` calls `len(text) // 4` without verifying if `text` is a valid string. If an LLM call fails, gets rate-limited, or returns `None`, and the framework tries to log the call via `log_call()`, a `TypeError: object of type 'NoneType' has no len()` is raised, crashing the entire pipeline.
*   **Destructive Scenario:** Triggered in automated test `test_token_tracker_null_input_crash`.
*   **Recommended Fix:**
    ```python
    def estimate_tokens(self, text: str) -> int:
        if not text or not isinstance(text, str):
            return 0
        return max(1, len(text) // 4)
    ```

### Bug 1.2: Read States JSON Corruption Crash (High)
*   **Severity:** High
*   **Component:** State Management / File I/O
*   **Affected File:** [execution.py](file:///e:/Mult_agent/agency/execution.py#L155-L165) (`get_unread_delta`)
*   **Description:** Unlike `update_read_state`, which wraps `json.loads(read_file(READ_STATES_FILE))` in a `try-except` block to handle empty or malformed files, `get_unread_delta` calls it directly without protection. If the JSON file is corrupted or empty (due to concurrent writes or interrupted writes), the framework throws a `JSONDecodeError` and halts.
*   **Destructive Scenario:** Triggered in automated test `test_get_unread_delta_corruption_crash`.
*   **Recommended Fix:** Wrap `json.loads` in a try-except block inside `get_unread_delta` and fall back to an empty dictionary.

### Bug 1.3: Jina AI Search Fallback Failure (Medium)
*   **Severity:** Medium
*   **Component:** Research Phase
*   **Affected File:** [api_bridge.py](file:///e:/Mult_agent/api_bridge.py#L209-L211) (`_call_jina_search`)
*   **Description:** The fallback logic from Jina AI Search to DuckDuckGo relies on checking:
    ```python
    if result_text and "CONTENT:" not in result_text and "ERROR:" in result_text:
    ```
    However, if the Jina API returns an HTTP 500 error, `res.text` is empty or does not contain the word `"ERROR:"`. Thus, the fallback is skipped, returning an empty research result to the coding agents instead of retrieving data via DuckDuckGo.
*   **Destructive Scenario:** Simulate a Jina search returning an empty HTTP 500 payload; the framework returns empty context.
*   **Recommended Fix:** Use explicit HTTP response status code checking and call the fallback on any non-200 Jina API response.

### Bug 1.4: Double Google Font Imports (Low)
*   **Severity:** Low (Performance Anti-pattern)
*   **Component:** Research Configuration
*   **Affected File:** [config.py](file:///e:/Mult_agent/config.py#L46-L55)
*   **Description:** The PM instruction forces agents to load Google Fonts using both a `<link>` in HTML and an `@import` statement in CSS. This causes double HTTP requests and blocks page rendering on luxury sites, violating high-fidelity standards.
*   **Destructive Scenario:** Code generated by agents fails PageSpeed audit due to redundant render-blocking assets.
*   **Recommended Fix:** Update the `CEO` system instructions in `config.py` to specify `<link>` tags only.

## Phase 2: Sandbox & Execution Bugs

### Bug 2.1: Timeout Error Message Inconsistency (Medium)
*   **Severity:** Medium
*   **Component:** Test Execution & Sandboxing
*   **Affected File:** [testing.py](file:///e:/Mult_agent/agency/testing.py#L163-L176) (`execute_in_sandbox`)
*   **Description:** The process execution sets a timeout parameter of `60` seconds (`timeout=60` on line 102). If a process times out, the exception handler prints:
    `"⏳ Process hit timeout (10s). Script failed to finish in time."`
    and returns:
    `"Error: Process timed out after 10 seconds."`
    This creates misleading logs since the process actually ran for 60 seconds.
*   **Destructive Scenario:** Triggered in automated test `test_execute_in_sandbox_timeout_message_inconsistency`.
*   **Recommended Fix:** Make the log messages dynamically output the actual timeout parameter used in the call or update the text to "60 seconds".

### Bug 2.2: Non-Portable Python Command Invocations (Medium)
*   **Severity:** Medium
*   **Component:** Environment Sandbox
*   **Affected File:** [testing.py](file:///e:/Mult_agent/agency/testing.py#L194) (`run_tests_safe`)
*   **Description:** The test runner spawns subprocesses using the hardcoded `"python"` executable:
    `subprocess.run(["python", test_script], ...)`
    If the system runs inside a Python virtual environment or Unix system where only `python3` or `sys.executable` points to the active Python path, this command fails with `FileNotFoundError` or runs tests in the wrong Python environment (without installed dependencies like flask).
*   **Destructive Scenario:** Running tests on a system with only `python3` configured on PATH.
*   **Recommended Fix:** Change `"python"` to `sys.executable` to ensure the current Python environment is inherited.

### Bug 2.3: Unbounded Infinite Loop Execution in Sandbox (High)
*   **Severity:** High
*   **Component:** Sandboxing Security
*   **Affected File:** [testing.py](file:///e:/Mult_agent/agency/testing.py#L98-L104) (`execute_in_sandbox`)
*   **Description:** The timeout parameter in `execute_in_sandbox` is set to 60 seconds. If an agent drafts an infinite loop inside a Flask web app startup sequence, the sandbox process will hang for a full minute per attempt, taking 3 minutes total over 3 attempts and wasting significant resources.
*   **Destructive Scenario:** Launching `execute_in_sandbox` with a python script containing `while True: pass`.
*   **Recommended Fix:** Reduce the timeout to a tighter window (e.g. 10 or 15 seconds) for initial smoke tests.

## Phase 3: Triage & Routing Bugs

### Bug 3.1: Greedy Regex Handoff JSON Parser Bug (High)
*   **Severity:** High
*   **Component:** Agent Orchestration / Communication
*   **Affected File:** [execution.py](file:///e:/Mult_agent/agency/execution.py#L120) (`extract_handoff`)
*   **Description:** The regex used to find the JSON block is greedy: `re.findall(r'\{.*\}', raw_output, re.DOTALL)`. If an agent outputs multiple separate JSON objects (e.g., in a log block and a handoff block), the regex matches from the first `{` of the first block to the last `}` of the final block. This includes all conversational text in between, making it invalid JSON.
*   **Destructive Scenario:** Triggered in automated test `test_extract_handoff_greedy_regex_bug`.
*   **Recommended Fix:** Use non-greedy regex matching: `re.findall(r'\{.*?\}', raw_output, re.DOTALL)`.

### Bug 3.2: Ollama Thinking Tag Triage Vulnerability (Medium)
*   **Severity:** Medium
*   **Component:** Issue Classification
*   **Affected File:** [triage.py](file:///e:/Mult_agent/agency/triage.py#L67-L72) (`ollama_triage_issue`)
*   **Description:** The triage handler removes thinking tags using `re.sub(r'<think>.*?</think>', '', raw, flags=re.DOTALL)`. If the model output is truncated or the model omits the closing `</think>` tag, the regex fails to match. The triage system will search the raw thinking trace for keywords like `"code_issue"` or `"design_issue"`, resulting in a false-positive classification.
*   **Destructive Scenario:** Ollama outputs a thinking log containing the word "design_issue" but gets truncated before producing the final token; the system classifies it as a design issue.
*   **Recommended Fix:** Change the regex to match open-ended thinking blocks: `re.sub(r'<think>.*?(?:</think>|$)', '', raw, flags=re.DOTALL)`.

### Bug 3.3: Empty Input Hang in Manual Triage Fallback (Low)
*   **Severity:** Low
*   **Component:** CLI Interaction
*   **Affected File:** [triage.py](file:///e:/Mult_agent/agency/triage.py#L88) (`ask_user_triage`)
*   **Description:** If Ollama is down and the user hits Enter (empty string) on the classification choice, `mapping.get(choice)` returns `None`, which defaults to `"code_issue"`. While it doesn't crash, it silently routes UI/UX issues to the RnD pipeline.
*   **Destructive Scenario:** Triggered by entering an empty string on manual triage input.
*   **Recommended Fix:** Loop input collection until a valid option (1, 2, or 3) is selected.

## Phase 4: Patching & AST Bugs

### Bug 4.1: Broken Test Function Extraction Logic (Medium)
*   **Severity:** Medium
*   **Component:** Targeted Patching
*   **Affected File:** [patching.py](file:///e:/Mult_agent/agency/patching.py#L135-L141) (`identify_broken_functions`)
*   **Description:** The regex `re.sub(r'^test_\d*_?', '', test_name)` transforms test names like `test_home_page_status_code` into `home_page_status_code`. It then looks for a function named `home_page_status_code` in the production file. Since the actual function in `final_code.py` is named `index`, the check fails, causing the framework to fall back to "broad patch mode" and modify the entire file.
*   **Destructive Scenario:** A test case failures log is parsed for function names; target matching fails.
*   **Recommended Fix:** Map test names to production functions using metadata or parse AST call expressions in the test file.

### Bug 4.2: AST Syntax Error Crash in Patch Extraction (Medium)
*   **Severity:** Medium
*   **Component:** Patching / AST
*   **Affected File:** [patching.py](file:///e:/Mult_agent/agency/patching.py#L69-L71) (`extract_function_block`)
*   **Description:** If a production file contains a syntax error (which frequently happens when coders output incomplete files), `ast.parse` throws a `SyntaxError`. While caught, it prints an error and returns `""`, which causes the patching system to fail the cycle completely instead of trying broad patching.
*   **Destructive Scenario:** Triggered in automated test `test_extract_function_block_syntax_error`.
*   **Recommended Fix:** Add a fallback regex function extractor when AST parsing fails due to syntax errors.
